47 small agent checks are more useful than 1 dramatic local-model manifesto.

That is my read on Ollama v0.23.1 adding Gemma 4 MTP speculative decoding for the MLX runner. The release claim is narrow and useful: over 2x speed increase for Gemma 4 31B on coding tasks on Macs.

I would not frame this as "local beats cloud." That is the wrong buying question for operators.

The better question is: which parts of the work are too cheap, too private, or too repetitive to send straight to the best frontier model?

For gofreddy and the harness work around it, I would put local coding models in 4 places:

1. Preflight before the agent run

Turn a messy bug note into a reproducible checklist. Check whether the file names exist. Shrink a failing trace before a more expensive model reads it.

2. Private scratch cleanup

Some material is not strategically sensitive, but still does not need to leave the machine. A local pass can redact client-adjacent context or summarize raw notes before the main run.

3. Cheap disagreement

I like using weaker or cheaper models to ask annoying questions. What assumption is missing? What test would fail first? What did the summary ignore?

4. Offline harness testing

When the point is to test the loop, not get the final answer, latency and cost matter more than brilliance. A faster local model makes repeated dry runs less painful.

The operating shift is not sovereignty theatre. It is budget discipline.

If the local model makes cleanup feel free, the frontier call gets a cleaner prompt, a smaller context, and a better-defined job. That is where the speed bump becomes a business tool instead of a benchmark headline.

# A1 — Root-cause for per-axis score collapse

## Verdict: Hypothesis 3 (aggregator back-fill), with prompt extension required

## Data flow

1. `autoresearch/archive/v006/scripts/evaluate_session.py:340` calls
   `_invoke_external_critique(critique_requests)`, where `critique_requests`
   contains one entry per rubric (criterion_id + rubric_prompt + output_text).
2. `_invoke_external_critique` (lines 226-240) shells out to
   `freddy evaluate critique -`, passing the full list of criteria.
3. The freddy CLI handler is `cli/freddy/commands/evaluate.py:_handle_legacy_batch_critique`
   (lines 128-226). Despite receiving N criteria, it:
   - Concatenates **all** rubrics into a single `session_goal` (lines 181-187)
   - Posts ONE request to `/invoke/critique` (line 191)
   - Parses ONE `overall` verdict and maps it via `_VERDICT_TO_SCORE = {pass=1.0, rework=0.5, fail=0.0}` (line 207)
   - **Broadcasts that single score across every criterion** (lines 215-225)
4. evaluate_session.py copies `normalized_score` straight into `score` (line 368).

## Why this exists

In-line comment at lines 137-139:
> "Granularity is lost (no per-criterion scoring) but session completes,
> structural gates pass, and we avoid timeout cascades. A future
> /invoke/critique-batch endpoint could restore per-criterion scoring."

A timeout cascade existed in pre-batch mode (one call per criterion × ~10 criteria × 300s timeout exhausted the judge service). The batched call was the deliberate (and never-reverted) tradeoff.

## Judge contract today

`judges/session/prompts/critique.md` returns only:
```json
{ "overall": "pass" | "rework" | "fail", "confidence": ..., "issues": [...], "rationale": "..." }
```

No per-criterion granularity is requested or produced.

## Fix (chosen approach: Option A — extend critique contract)

Change the critique JSON schema to ALSO return `per_criterion[]`:
```json
{
  "overall": "...",
  "per_criterion": [
    {"criterion_id": "GEO-1", "verdict": "pass" | "rework" | "fail", "rationale": "..."}
  ],
  ...
}
```

- `judges/session/prompts/critique.md` — extend output schema, instruct judge to return one entry per criterion in `session_goal`.
- `cli/freddy/commands/evaluate.py:_handle_legacy_batch_critique` — when `per_criterion` is present, build per-criterion results from it. Fall back to broadcast when absent (graceful degradation if the judge model omits the field).

LOC estimate: ~60.

## Counter-evidence checked

- The variant_scorer at `judges/evolution/agents/variant_scorer.py` already supports `per_criterion[]` — verified at line 28-33 of `judges/evolution/prompts/scorer.md`. Pattern is already in the codebase.
- 4/188 KEEP files in v006 showed *some* axis variance (unique count = 2). Those are likely the few cases where the verdict was tied across rubrics but `_VERDICT_TO_SCORE` happened to differ between primary/secondary judges; OR they come from a different code path (variant_scorer rather than the critique bridge). Worth verifying after the fix lands.
